# PromptProxy 総合仕様書

## 1. システム概要

### 1.1 目的
PromptProxyは、強力なローカルAIコーディングツールである「Aider」と、ブラウザ上の強力なLLM（Gemini 1.5 ProやClaude 3.5 Sonnet等のWeb UI）をシームレスに中継（プロキシ）するためのGUIアプリケーションです。
Aiderからは「OpenAI API互換のローカルサーバー」として振る舞い、Web AIの巨大なコンテキストウィンドウと高度な推論能力をAiderに統合することを目的としています。

### 1.2 主な特徴
* **APIキー完全不要**: Webブラウザ上のLLMチャットUIで完結します。
* **巨大プロジェクト対応**: Aiderが生成したプロンプトを `context.xml` として物理ファイル化し、ブラウザへのドラッグ＆ドロップに対応することで、長文テキスト貼り付け時のフリーズを回避します。
* **自律的エージェント機能**: AIがコンテキスト不足を検知し、自ら追加ファイルを要求して再起動するオートメーション機能を搭載しています。
* **OS越境D&D**: Windows/MacとLinux(Wayland/X11)間で異なるドラッグ＆ドロップの仕様を内部で吸収しています。

---

## 2. システムアーキテクチャ

本システムは Tauri (Rust) + React (TypeScript) で構成されています。

### 2.1 フロントエンド (View層)
* **技術**: React, TypeScript, CSS (Vanilla)
* **役割**: ユーザーからの指示入力、設定の管理、OSへのドラッグ＆ドロップの提供、AIからの返答入力UI、およびAiderプロセスの標準出力のリアルタイム表示を行います。

### 2.2 バックエンド (Controller/Proxy層)
* **技術**: Rust, Tauri, Axum (Webサーバー), Tokio (非同期処理)
* **役割**:
  * `localhost:8080` にてOpenAI互換のモックAPIサーバー (`/v1/chat/completions`) を稼働。
  * Aiderの子プロセス管理（起動、引数構築、標準出力パイプ、Graceful Shutdown）。
  * Aiderから受信したJSONプロンプトのパースと、XMLファイル化。
  * ファイル追加要求 (`[ADD_FILE: ...]`) のインターセプトとプロセスの自動再起動。

## 3. 画面構成とUI仕様 (垂直パイプラインレイアウト)

モダンなIDEを意識した、状態を常時表示する高密度なレイアウトを採用しています。

### 3.1 画面レイアウト
画面は縦に3つの領域に分割されています。
1. **上部ヘッダー (ステータス領域)**: 現在のアプリ状態（Stepper）と、強制初期化用の「🔄 リロード」ボタンを配置。
2. **中央メイン領域**: ユーザーの操作やAIとのやり取りを行うメイン作業エリア。
3. **下部ターミナル領域**: Aiderプロセスの標準出力・標準エラー出力を常時表示する黒いターミナル（自動スクロール付き）。

### 3.2 メイン領域のカード構成
状態の切り替わりによる画面のちらつきを防ぐため、以下の3つのカードが縦に常時表示され、アクティブでないカードはグレーアウト（入力不可）されます。
* **[Card 1] ユーザー指示**: ディレクトリ、対象ファイル、モード(Edit/Ask)、指示内容、言語を指定。
* **[Card 2] LLMへの指示**: 生成された `context.xml` のドラッグ用アイコンと、LLMへのテキストプロンプト（コピーボタン付き）を表示。
* **[Card 3] LLMからの指示**: Web UIからコピーしたAIの返答を貼り付け、Aiderへ返却するテキストエリア。

### 3.3 設定画面 (⚙️ 設定)
メイン画面上部の設定ボタンから遷移し、以下の項目をカスタマイズ（`localStorage` に永続化）できます。
* **チャット言語**: Aiderの思考・返答言語（`--chat-language` オプションに連動）。
* **カスタムプロンプト切り替え**: 組み込みのデフォルトプロンプトと、ユーザー独自のプロンプト（Edit用/Ask用）を切り替え可能。

## 4. 状態遷移 (State Machine)

React側の `appState` により、アプリは以下の3つの状態を遷移します。

1. **`init` (初期状態)**:
   * ユーザーがディレクトリ、ファイル、指示を入力し「Aiderをバックグラウンドで実行」を押すのを待機。
2. **`idle` (Aider実行中)**:
   * Aiderのプロセスが起動し、ソースコードを読み込んでリポジトリマップ等を構築している状態。ターミナルにAiderのログが流れる。
3. **`pending` (LLMプロキシ応答待ち)**:
   * AiderがAPIリクエストを送信し、Rustがそれをパースして `context.xml` とプロンプトを生成した状態。
   * ユーザーがAIからの返答をテキストエリアに入力し、「Aiderに返す」を押すのを待機。返却後、再び `idle`（または終了）に戻る。

## 5. 主要機能・コアルール仕様

### 5.1 プロンプト生成とXML化
Aiderが生成した `messages` をRustでパースし、以下のように再構築します。
* `context.xml`: リポジトリ情報と対象ファイルのソースコード（Markdownのコードブロックを抽出したもの）を `<repository>` および `<file path="...">` タグで囲み、CDATAセクションでエスケープして `/tmp/` ディレクトリ等に保存。
* `prompt` (文字列): ユーザーの指示と、AIへのフォーマット指示（SEARCH/REPLACEブロックの要求など）を結合したもの。

### 5.2 操作モード (Edit / Ask) の切り替え
* **Edit モード (コード修正)**: Aiderへの指示はそのまま渡され、AIには「挨拶を省き、コードブロック内の SEARCH/REPLACE 構文のみを出力する」ことを厳格に要求します。
* **Ask モード (質問)**: Aiderへの指示の先頭に自動で `/ask ` コマンドを付与して起動し、AIには「自然なテキストで回答する」ことを要求します。

### 5.3 自律的ファイル追加・再起動機能 (`[ADD_FILE]`)
Editモード時、AIへのシステムプロンプトに「情報が不足している場合は `[ADD_FILE: ファイルパス]` のみを出力せよ」と指示しています。
1. ユーザーが `[ADD_FILE: src/xxx.ts]` をアプリにペーストして返す。
2. Rustバックエンドが正規表現でこれを検知。
3. 現在のAiderプロセスにダミーの返答（"Understood..."）を返して正常終了させる。
4. 対象ファイルリスト (`files`) に要求されたファイルを追加し、自動的にAiderプロセスを再起動する。

### 5.4 OS非依存のドラッグ＆ドロップ機能
`context.xml` をブラウザにドロップする際、OSの仕様に合わせて処理を分岐します。
* **Linux環境 (Wayland/X11)**: WebViewのデフォルトドラッグイベント（`e.dataTransfer`）を利用し、`text/uri-list` として純粋な絶対パスを渡すことで、ファイルマネージャーやEmacs等へのドロップを可能にします。
* **Mac/Windows環境**: `@crabnebula/tauri-plugin-drag` ネイティブAPIを使用し、OSのドラッグプロトコルを呼び出します。

### 5.5 Aiderプロセスの安全な終了 (Graceful Shutdown)
アプリを閉じる（ウィンドウを閉じる）際、Aiderプロセスがゾンビ化して `.aider.chat.history.md` 等のロックが残らないよう、以下の手順で終了します。
1. プロセスIDに対して `SIGINT` (Ctrl+C 相当) を送信。
2. 最大2秒間、プロセスの正常終了をポーリング待機。
3. タイムアウトした場合は `SIGKILL` で強制終了。
